## Q01 · RAG 是什么 / 为什么需要

### 🎯 RAG 解决的核心问题

模型在运行时**需要知道什么**。

RAG 做法：先从外部文档库**检索**，再带着检索到的证据**生成**答案。

适合：**FAQ、企业知识库、规章制度、产品文档、法律/医学资料问答**这类"答案要以文档为准"的任务。

### 📖 标准流水线

```
用户 Query
   ↓
[1] Query 理解 / 改写
   ↓
[2] 检索相关文档（多路召回）
   ↓
[3] 重排序（Reranking）
   ↓
[4] 拼接 Prompt（context + query）
   ↓
[5] LLM 生成答案
```

### 🪤 追问

- **Q：RAG 比直接微调好在哪？**
  A：
  1. **可更新**：换文档库就能换知识，不用重训
  2. **可追溯**：能告诉你证据在哪
  3. **省成本**：不需要为每个领域训一个模型
  4. **隐私**：知识库可本地部署

## Q02 · 索引构建

### 📖 整体流程

```
原始文档 → 解析(PDF/HTML/...) → 清洗 → 分块(Chunking) → 向量化(Embedding) → 存入向量数据库
```

### 📖 文本切分（Chunking）

文档通常太长，不能整篇直接检索，要切成小块。

| 策略 | 描述 | 适用 |
|---|---|---|
| **固定大小** | 按 token 数切分 | 简单文本 |
| **句子级** | 按句号切 | 精确检索 |
| **段落级** | 按段落切 | 保上下文 |
| **递归分割** | 按分隔符递归切 | 通用（**推荐**） |
| **语义分割** | 按语义相似度切 | 高质量 |

**Chunk 大小建议**：

- 通常 **256-1024 tokens**
- 加 **10-20% overlap**（避免语义被切断）
- 根据下游任务调整

### 📖 向量化（Embedding）

把每个 chunk 编码成向量，**相似语义在向量空间中接近**。

#### 常用 Embedding 模型

| 模型 | 维度 | 特点 |
|---|---|---|
| OpenAI text-embedding-3-large | 3072 | 商业最强 |
| **BGE-M3** | 1024 | 多语言、多粒度（**国内首选**） |
| E5-Mistral-7B-Instruct | 4096 | 指令增强 |
| GTE-Qwen2 | 1024 | 中英双语优秀 |

### 📖 向量数据库

存储 + 索引 embedding，支持快速近邻搜索：

- **FAISS**：Meta 开源，快但要自己加持久化
- **Milvus / Qdrant / Weaviate**：开源向量数据库
- **Chroma**：轻量，本地用
- **Pinecone**：商业 SaaS

### 🪤 追问

- **Q：Chunk 越小越好吗？**
  A：不一定。太小丢失上下文，太大稀释语义。需要根据任务实验，常见 512 tokens 起步。

## Q03 · 查询改写

完整的 RAG 在线流程不是直接检索，更像：
$$
\text{Query} \to \text{Query 改写} \to \text{召回} \to \text{重排} \to \text{上下文构建}
$$

### 1) Query 规范化

清洗原始问题：

- 统一大小写
- 去多余停用词
- 标准化术语
- 缩写展开
- 拼写纠正

### 2) Query 改写

把自然语言改写为更适合检索的 query：

- 关键词更明确
- 语义更完整
- 专业术语更规范

### 3) Query 扩展

太短的 query 扩展出**相关词、同义词、上下位词**。

### 4) Query 拆解

复杂问题不能一次检索完成，**拆解成多个子问题**。
例：「2024 年中国 GDP 和美国 GDP 分别是多少？」→ 拆成 2 个子问题。

### 📖 高级方案

- **HyDE**（Hypothetical Document Embeddings）：让 LLM 先**假设性地写一个回答**，再用这个回答去做向量检索（让检索目标语义更接近文档语义）
- **Multi-Query**：让 LLM 改写出多个等价 query，并行检索后合并

## Q04 · 召回策略

### 📊 三大检索方式

| 方法 | 原理 | 优点 | 缺点 |
|---|---|---|---|
| **向量检索** | 语义相似度 | 理解语义 | 精确匹配差 |
| **关键词（BM25）** | 词频统计 | 精确匹配好 | 不理解语义 |
| **混合检索** | **两者结合** | **互补** | 需要融合 |

### 📖 混合检索的融合方式

最常用 **RRF（Reciprocal Rank Fusion）**：
$$
\text{score}(d) = \sum_i \frac{1}{k + \text{rank}_i(d)}
$$

### 📖 多路召回

从多个视角召回，最后合并：

- 原始 query
- 改写 query
- 扩展 query
- 子问题 query

### 📖 召回数量

一般初召回 **Top-20/50/100**，后续 rerank 选 **Top-3/5**。

## Q05 · 重排序

### 🎯 为什么需要 Reranking？

向量检索本质是**看"谁最像"**，而不是**"谁最能回答"**。
重排序用 **Cross-Encoder**：query 和 doc 一起编码，精度更高。

### 📖 双塔 vs 单塔

| | Bi-Encoder（双塔） | Cross-Encoder（单塔） |
|---|---|---|
| 编码 | query 和 doc 分别编码 | 一起编码 |
| 速度 | 快（向量可预存） | 慢（每对都要算） |
| 精度 | 中 | **高** |
| 用途 | 一阶段召回 | 二阶段精排 |

### 📖 常用 Reranker

- **BGE-Reranker-v2**（国内首选，开源）
- **Cohere Reranker**（商业）
- **Cross-Encoder/ms-marco**（学术基线）

### 🪤 追问

- **Q：为什么不直接用 cross-encoder 跑全库检索？**
  A：太慢。cross-encoder 每次要重新编码（query, doc）对，全库 N 万条要算 N 万次 forward。

## Q06 · RAG 评测指标

| 维度 | 指标 | 衡量 |
|---|---|---|
| **检索质量** | Recall@K, MRR, NDCG | 检索准确性 |
| **生成忠实度** | Faithfulness | 是否基于检索内容 |
| **答案相关性** | Answer Relevancy | 回答是否切题 |
| **上下文精确度** | Context Precision | 检索结果相关性 |

### 📖 关键指标解释

- **Recall@K**：Top-K 中有多少是相关文档
- **MRR**（Mean Reciprocal Rank）：第一个相关文档排名的倒数均值
- **NDCG**：考虑排序位置的加权指标
- **Faithfulness**：答案中的事实是否都有 context 支撑（防幻觉）

### 📖 评测框架

- **RAGAS**：自动化 RAG 评测，支持上述四大指标
- **TruLens**：可视化 RAG 调试

## Q07 · 高级 RAG

### 📖 Agentic RAG

让 **Agent 决定何时检索、检索什么、如何使用结果**：

```
用户查询
  → Agent 判断：需要检索吗？
    → 是：制定检索策略 → 检索 → 判断结果是否充分
      → 不充分：改写查询重新检索
      → 充分：生成回答
    → 否：直接回答
```

### 📖 Graph RAG

利用**知识图谱**增强检索：

- 实体识别 → 构建 KG → 图检索 + 向量检索
- 适合**多跳推理和关系型查询**

例：「张三的同事的导师是谁？」需要跨多个三元组才能回答。

### 📖 Self-RAG

让模型自己学会**何时引用、何时直接答**，并对自己的输出做反思。

## Q08 · RAG vs 长上下文 vs 微调

| 维度 | RAG | 长上下文 | 微调 |
|---|---|---|---|
| **更新成本** | 低（换库） | 中（重传） | 高（重训） |
| **可追溯** | ✅ 有引用 | ✅ 有原文 | ❌ |
| **训练成本** | 0 | 0 | 高 |
| **推理成本** | 中 | 高（KV Cache 大） | 低 |
| **新知识** | ✅ | ✅ | ❌（需重训） |
| **风格定制** | ❌ | ❌ | ✅ |
| **指令遵循** | ❌ | 中 | ✅ |

**通常组合用**：

- 风格 + 指令 → SFT
- 知识 → RAG
- 短期上下文 → 长上下文窗口

### 🪤 追问

- **Q：长上下文模型出了，RAG 是不是被淘汰了？**
  A：不会。RAG 在以下场景仍优：
  1. 知识库远超模型上下文（百万文档）
  2. 需要**可追溯证据**（医疗、法律）
  3. **隐私**：私有数据不进 LLM 提示
  4. **成本**：长上下文每次调用都很贵
